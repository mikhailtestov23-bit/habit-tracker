# Инструкции для создания трекера привычек

## Цель проекта

Создать веб-приложение для трекинга привычек, где пользователь может добавлять любые привычки, отмечать их выполнение с гибкой периодичностью, хранить историю событий, импортировать и экспортировать данные, получать напоминания и красивые ачивки за прогресс.

## Рекомендуемый стек

Для первой версии:

- Frontend и backend: Next.js.
- Язык: TypeScript.
- База данных: PostgreSQL.
- ORM: Prisma или Drizzle.
- Авторизация: Supabase Auth или NextAuth.
- Хостинг приложения: Vercel.
- Хостинг базы: Supabase.
- Стили: Tailwind CSS.
- Ачивки и микроанимации: CSS-анимации, Framer Motion или Motion.

Почему так:

- Next.js позволяет держать интерфейс, API и серверную логику в одном проекте.
- PostgreSQL хорошо подходит для событийной истории привычек.
- Supabase дает управляемую PostgreSQL-базу, авторизацию, экспорт/импорт и возможность фоновых задач.
- Vercel хорошо подходит для деплоя Next.js и поддерживает cron-задачи для регулярных проверок.

## Основные сущности

### User

Пользователь приложения.

Поля:

- `id`: UUID.
- `email`: строка.
- `name`: строка, опционально.
- `timezone`: строка, например `Europe/London`.
- `created_at`: дата создания.
- `updated_at`: дата обновления.

### Habit

Привычка, которую пользователь хочет отслеживать.

Поля:

- `id`: UUID.
- `user_id`: UUID владельца.
- `title`: название привычки, например `Выпить воду`.
- `description`: описание, опционально.
- `color`: цвет привычки в интерфейсе.
- `icon`: имя иконки, например `droplet`, `book`, `dumbbell`.
- `frequency_type`: `hourly`, `daily`, `weekly`, `custom`.
- `target_count`: сколько раз нужно выполнить за период.
- `period_interval`: число интервалов, например каждые `2` дня.
- `period_unit`: `hour`, `day`, `week`.
- `weekdays`: массив чисел `1-7`, если привычка должна выполняться в конкретные дни недели.
- `starts_at`: дата начала.
- `ends_at`: дата окончания, опционально.
- `is_active`: активна ли привычка.
- `created_at`: дата создания.
- `updated_at`: дата обновления.

Примеры:

- Каждый день: `frequency_type = daily`, `target_count = 1`, `period_interval = 1`, `period_unit = day`.
- Каждые 3 часа: `frequency_type = hourly`, `target_count = 1`, `period_interval = 3`, `period_unit = hour`.
- 4 раза в неделю: `frequency_type = weekly`, `target_count = 4`, `period_interval = 1`, `period_unit = week`.
- По понедельникам, средам и пятницам: `frequency_type = custom`, `weekdays = [1, 3, 5]`.

### HabitEvent

Факт выполнения привычки. Это главная история трекера.

Поля:

- `id`: UUID.
- `habit_id`: UUID привычки.
- `user_id`: UUID пользователя.
- `occurred_at`: дата и время события.
- `value`: число, по умолчанию `1`.
- `note`: заметка, опционально.
- `source`: `manual`, `import`, `reminder`, `automation`.
- `created_at`: дата создания записи.
- `updated_at`: дата обновления.

Важно: не хранить прогресс только агрегатами вроде `streak = 10`. История должна строиться из событий `HabitEvent`, а серии, проценты и графики лучше пересчитывать из событий или хранить в отдельной таблице-кэше.

### Reminder

Напоминание для привычки.

Поля:

- `id`: UUID.
- `habit_id`: UUID привычки.
- `user_id`: UUID пользователя.
- `channel`: `push`, `email`, `in_app`.
- `time_of_day`: время, например `09:00`.
- `timezone`: часовой пояс.
- `weekdays`: массив чисел `1-7`, опционально.
- `is_enabled`: включено ли напоминание.
- `last_sent_at`: когда отправлялось в последний раз.
- `created_at`: дата создания.
- `updated_at`: дата обновления.

### Achievement

Описание ачивки.

Поля:

- `id`: UUID.
- `code`: стабильный код, например `first_checkin`, `seven_day_streak`.
- `title`: название.
- `description`: описание.
- `icon`: иконка или путь до изображения.
- `rarity`: `common`, `rare`, `epic`, `legendary`.
- `condition_type`: тип условия, например `event_count`, `streak`, `perfect_week`.
- `condition_value`: число, например `7`.
- `created_at`: дата создания.

### UserAchievement

Полученная пользователем ачивка.

Поля:

- `id`: UUID.
- `user_id`: UUID пользователя.
- `achievement_id`: UUID ачивки.
- `habit_id`: UUID привычки, опционально.
- `unlocked_at`: дата получения.

## Минимальный набор экранов

### Главный экран

Должен показывать:

- список привычек на сегодня;
- быстрые кнопки отметки выполнения;
- текущий прогресс по каждой привычке;
- короткую историю за последние 7 дней;
- заметную, но не мешающую анимацию при выполнении.

### Экран привычки

Должен показывать:

- название, описание, цвет, иконку;
- календарь или heatmap выполнения;
- список событий;
- streak;
- статистику за неделю, месяц и все время;
- настройки периодичности;
- настройки напоминаний.

### Экран добавления/редактирования привычки

Поля:

- название;
- описание;
- цвет;
- иконка;
- периодичность;
- цель за период;
- дата начала;
- дата окончания;
- напоминания.

### Экран импорта/экспорта

Должен позволять:

- экспортировать все данные пользователя в JSON;
- экспортировать события в CSV;
- импортировать JSON обратно;
- перед импортом показать предпросмотр: сколько привычек, событий и напоминаний будет добавлено;
- сообщить об ошибках в строках/записях без потери уже существующих данных.

## Логика периодичности

Нужно поддержать три основных режима:

### По часам

Пример: `делать разминку каждые 2 часа`.

Логика:

- период считается от `starts_at`;
- выполнение засчитывается в окно времени;
- если цель `target_count = 1`, достаточно одного события в окне.

### По дням

Пример: `читать 20 минут каждый день`.

Логика:

- день считается в часовом поясе пользователя;
- прогресс дня равен сумме `HabitEvent.value` за день;
- день выполнен, если сумма >= `target_count`.

### По неделям

Пример: `тренироваться 3 раза в неделю`.

Логика:

- неделя считается в часовом поясе пользователя;
- по умолчанию неделя начинается в понедельник;
- неделя выполнена, если сумма событий за неделю >= `target_count`.

## Импорт и экспорт данных

### Рекомендуемый формат экспорта JSON

```json
{
  "schema_version": 1,
  "exported_at": "2026-07-03T12:00:00.000Z",
  "user": {
    "timezone": "Europe/London"
  },
  "habits": [
    {
      "id": "b7a3a6e3-8d97-4f34-8b93-7b2d78d8e620",
      "title": "Выпить воду",
      "description": "Стакан воды утром",
      "color": "#3B82F6",
      "icon": "droplet",
      "frequency_type": "daily",
      "target_count": 1,
      "period_interval": 1,
      "period_unit": "day",
      "weekdays": null,
      "starts_at": "2026-07-01T00:00:00.000Z",
      "ends_at": null,
      "is_active": true
    }
  ],
  "events": [
    {
      "id": "08fbf6f1-f00f-41ff-93d8-9463f305e90a",
      "habit_id": "b7a3a6e3-8d97-4f34-8b93-7b2d78d8e620",
      "occurred_at": "2026-07-03T08:10:00.000Z",
      "value": 1,
      "note": "После завтрака",
      "source": "manual"
    }
  ],
  "reminders": [
    {
      "habit_id": "b7a3a6e3-8d97-4f34-8b93-7b2d78d8e620",
      "channel": "push",
      "time_of_day": "09:00",
      "timezone": "Europe/London",
      "weekdays": [1, 2, 3, 4, 5],
      "is_enabled": true
    }
  ]
}
```

### Правила для импорта JSON

- `schema_version` обязателен.
- `habits` и `events` обязательны, даже если массивы пустые.
- `id` можно передавать из старого экспорта. Если `id` отсутствует, приложение должно создать новый UUID.
- `habit_id` в событиях должен ссылаться на привычку из импортируемого файла или на уже существующую привычку пользователя.
- Все даты передавать в ISO 8601 UTC, например `2026-07-03T08:10:00.000Z`.
- Часовой пояс хранить отдельно в формате IANA, например `Europe/London`.
- При конфликте `id` не перезаписывать существующие данные молча. Лучше показать режимы: `пропустить`, `создать копию`, `обновить`.
- Импорт должен быть идемпотентным: повторный импорт одного файла не должен создавать дубли, если `id` совпадают.

### Рекомендуемый CSV для событий

CSV подходит для простого импорта истории выполнений.

Колонки:

```csv
habit_title,habit_id,occurred_at,value,note,source
"Выпить воду","b7a3a6e3-8d97-4f34-8b93-7b2d78d8e620","2026-07-03T08:10:00.000Z",1,"После завтрака","manual"
```

Правила:

- `occurred_at` обязателен.
- Достаточно одного из полей: `habit_id` или `habit_title`.
- Если указан только `habit_title`, приложение должно найти привычку по названию. Если найдено несколько совпадений, строка должна попасть в ошибки предпросмотра.
- `value` по умолчанию равен `1`.
- `source` по умолчанию равен `import`.

## Напоминания

Напоминания можно делать поэтапно.

### MVP

- Внутри приложения показывать список привычек, которые нужно выполнить сегодня.
- Использовать локальные браузерные уведомления, если пользователь разрешил их.

### Версия 2

- Добавить push-уведомления через Web Push.
- Добавить email-напоминания.
- Раз в несколько минут запускать фоновую задачу, которая ищет напоминания, подходящие по времени и часовому поясу.

### Важные правила

- Не отправлять повторное напоминание, если `last_sent_at` уже попадает в текущий временной слот.
- Всегда учитывать `timezone` пользователя.
- Пользователь должен иметь возможность выключить напоминания полностью или для конкретной привычки.

## Ачивки

Ачивка должна быть визуально приятной и ощущаться как маленькая награда, а не как обычный toast.

Примеры ачивок:

- `Первый шаг`: первая отметка любой привычки.
- `Три дня подряд`: streak 3 дня.
- `Неделя в огне`: streak 7 дней.
- `Идеальная неделя`: все запланированные привычки выполнены за неделю.
- `Возвращение`: привычка выполнена после паузы 7+ дней.
- `Мастер ритма`: 100 выполнений одной привычки.

Поведение:

- после выполнения события backend проверяет условия ачивок;
- если ачивка новая, создается `UserAchievement`;
- frontend показывает полноэкранную или модальную анимацию;
- ачивку можно посмотреть позже в профиле.

Визуальные требования:

- крупная иконка или медаль;
- цвет зависит от редкости;
- короткий заголовок;
- мягкая анимация появления;
- кнопка закрытия;
- отсутствие мигания и агрессивных эффектов.

## API

Минимальные endpoint-ы:

- `GET /api/habits` - список привычек.
- `POST /api/habits` - создать привычку.
- `PATCH /api/habits/:id` - обновить привычку.
- `DELETE /api/habits/:id` - архивировать или удалить привычку.
- `GET /api/habits/:id/events` - события привычки.
- `POST /api/habits/:id/events` - отметить выполнение.
- `DELETE /api/events/:id` - удалить ошибочную отметку.
- `GET /api/export` - экспорт JSON.
- `POST /api/import/preview` - предпросмотр импорта.
- `POST /api/import/commit` - применить импорт.
- `GET /api/achievements` - полученные ачивки.
- `POST /api/reminders/test` - тестовое напоминание.

## Валидация

Обязательные проверки:

- название привычки не пустое;
- `target_count` больше 0;
- `period_interval` больше 0;
- `period_unit` соответствует типу периодичности;
- даты валидны;
- событие не может принадлежать чужой привычке;
- импорт не может создать данные для другого пользователя;
- часовой пояс должен быть валидным IANA timezone.

## Безопасность

- Все пользовательские данные должны быть привязаны к `user_id`.
- Пользователь не должен видеть и менять привычки другого пользователя.
- Для Supabase включить Row Level Security.
- Для API проверять авторизацию на каждом запросе.
- Импорт выполнять только для текущего пользователя.
- Экспорт отдавать только владельцу данных.

## Тесты

Нужно покрыть:

- расчет выполнения за день;
- расчет выполнения за неделю;
- привычки с периодом по часам;

## Рекомендация по деплою

Оптимальный вариант для этого проекта: Vercel + Supabase.

Схема:

- Next.js-приложение деплоить на Vercel.
- PostgreSQL, авторизацию и хранение данных держать в Supabase.
- Для MVP напоминаний использовать локальные уведомления в браузере.
- Для серверных напоминаний использовать Vercel Cron Jobs или Supabase scheduled Edge Functions.

Почему это оптимально:

- мало DevOps-работы;
- удобно стартовать бесплатно или недорого;
- PostgreSQL не привязан к самописному серверу;
- проект легко масштабировать;
- экспорт данных можно строить поверх обычных SQL-запросов;
- при необходимости можно переехать на другой PostgreSQL-хостинг.

Альтернативы:

- Railway или Render: удобны, если нужен отдельный backend-сервер с долгоживущими задачами.
- Fly.io: хорош, если хочется больше контроля над регионом и инфраструктурой.
- Firebase: быстро для MVP, но хуже подходит, если нужна реляционная событийная модель и удобный SQL-экспорт.

Итоговая рекомендация: начинать с Vercel + Supabase, а отдельный backend добавлять только если напоминания, очереди или интеграции станут сложнее возможностей cron-задач и Edge Functions.
